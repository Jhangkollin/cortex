# cortex-web

Next.js 16 (App Router) dashboard. Talks to `cortex-api`.

## Quick start

```bash
npm install
cp .env.example .env.local  # set GOOGLE_CLIENT_ID / SECRET
npm run dev                  # opens http://localhost:3000
```

## Common commands

```bash
npm run dev                  # dev server (Turbopack)
npm run build                # production build (standalone output)
npm run lint                 # ESLint
npm run type-check           # tsc --noEmit
npm test                     # vitest

# Regen the typed API client from cortex-api OpenAPI
npm run generate-client      # writes to src/lib/api-client/generated/
```

## Conventions

- Server Components by default; mark `"use client"` only when needed
- Data fetching: TanStack Query options generated from OpenAPI; `prefetchQuery` + `<HydrationBoundary>` in RSC for instant first paint
- Auth: NextAuth + Google OAuth restricted to `@mlytics.com`
- Frontend rendering gated by `permissions.ts` active-context capability matrix

## Tests

Component tests run under **vitest + jsdom** with `@testing-library/react`.

```bash
npm test           # one-shot run (CI + pre-commit)
npm run test:watch # incremental, re-runs on file save
```

Layout:

```
web/
├── vitest.config.ts                       # jsdom env, @/* alias, setup file
├── tests/
│   ├── setup.ts                           # jest-dom matchers + module mocks
│   └── helpers/
│       ├── render-with-session.tsx        # render(...) wrapped in MockSessionProvider
│       └── session-mock-state.ts          # mutable next-auth + router holders
└── src/
    └── **/__tests__/*.test.tsx            # co-located component tests
```

Patterns:

- **Module mocks live in `tests/setup.ts`.** `next-auth/react` and
  `next/navigation` are stubbed there so every test file inherits the
  same fixture surface — drive state via `setMockSession(...)` and
  assert on `router.replace.mock.calls`.
- **Test the boundary, not the internals.** Assertions target observable
  side effects (`router.replace` calls, DOM presence) — never inspect
  `MockSessionProvider`'s context value directly.
- **Don't use `mock.patch`-style monkey-patching.** Per CLAUDE.md, mock at
  the module boundary with `vi.mock`. The component-under-test's imports
  resolve to the mocks transparently.

The current regression anchors are the server-side gate test
(`src/components/auth/__tests__/onboarding-gate.test.tsx`), which locks
down the `OnboardingGate` decision ladder including the dev-bypass and
error branches, and the sign-out test
(`src/components/auth/__tests__/sign-out-button.test.tsx`), which asserts
that clicking the control performs **both** the mock-session clear and
the NextAuth `signOut({ callbackUrl: "/signin" })`.
