# Cortex Go-Live Design — `cortex.mlytics.com` (PROD) / `cortex.mlytics.co` (UAT)

**Status:** Draft (sections being filled in collaboratively).
**Date:** 2026-05-11.
**Owner:** @okis.chuang.
**Scope:** Take the green-field `cortex/` web + api app from local-only to live on EKS in UAT, then PROD, using the existing `data-platform-helm-charts/` and `data-platform-iac/` repos.

## Locked-in decisions (from brainstorming)

1. **Sequencing:** UAT first → bake → PROD. Helm chart authoring and IaC modules done in parallel (Option B in brainstorm).
2. **Domain layout:** Single public host per env, web-only at the edge. `cortex-api` stays in-cluster. Wildcard ACM certs provisioned up-front so future `api.cortex…`, `admin.cortex…`, etc. are free.
3. **Datastores:** New, dedicated AWS RDS Postgres per env, provisioned by Terraform under the new `agents/` domain. **Redis is deferred** — cortex-api uses in-process LRU+TTL for the 60s membership cache, gated by `CORE_MEMBERSHIP_CACHE_BACKEND=memory|redis` so we can swap to managed Redis later without rework.
4. **IaC domain naming:** New sibling domain `agents/` introduced in `data-platform-iac` for cortex-era infra. Existing `aigc/` left untouched (it still owns `agent-will-smith`). Migration of `aigc/*` → `agents/*` is a separate, scheduled exercise — not part of go-live. Cortex ECR paths therefore become `agents/cortex-api` and `agents/cortex-web`, requiring a one-line update to `cortex/.github/workflows/deploy.yml`.
5. **DNS:** **Cloudflare-managed** (external registrar). Terraform outputs ACM validation CNAMEs and the final ALB CNAME target; a human pastes them into Cloudflare. **Public CNAMEs MUST be set to DNS-only (grey cloud), not Proxied (orange cloud)** — proxied records break ALB SNI against the ACM cert. Workarounds (Cloudflare Origin CA, Full-Strict mode) are out of scope for go-live; revisit later if Cloudflare proxying becomes desirable.
6. **Secrets:** Terraform creates empty AWS Secrets Manager entries with `lifecycle.ignore_changes = [secret_string]`. The RDS password is auto-generated by Terraform into its secret. Google OAuth (`client_id`, `client_secret`), `NEXTAUTH_SECRET`, and the Databricks PAT are pasted by a human via AWS console/CLI before the helm install.

## Section 1 — Architecture overview

### Runtime topology (per environment)

```
                External DNS (registrar)
                          │
              cortex.mlytics.{com|co}  CNAME →  <alb-dns-name>
                          │
                          ▼
                AWS ALB  (TLS terminated, ACM wildcard *.cortex.mlytics.{com|co})
                          │  HTTPS :443 → HTTP :3000
                          ▼
                K8s Service: cortex-web   (ClusterIP, port 3000)
                          │
                          ▼
                Pod: cortex-web (Next.js 16 standalone)
                          │  in-cluster fetch
                          │  http://cortex-api.cortex.svc.cluster.local:8000
                          ▼
                K8s Service: cortex-api   (ClusterIP, port 8000)
                          │
                          ▼
                Pod: cortex-api (FastAPI)
                          │
            ┌─────────────┼──────────────────┐
            ▼             ▼                  ▼
        RDS Postgres  Databricks       Secrets Manager
        (cortex DB)   SQL (analytics)  (via CSI driver)

        (Redis deferred — membership cache uses in-process LRU+TTL;
         see locked-in decision #3.)
```

### Component placement

| Concern | Decision | Notes |
|---|---|---|
| Container images | ECR repos `agents/cortex-api`, `agents/cortex-web` per AWS account | New `agents/` IaC domain (sibling of existing `aigc/`). `cortex/.github/workflows/deploy.yml` ECR_REPO_API/WEB env vars updated as part of cutover. |
| Namespace | `cortex` (new, dedicated) | Reusing `agent` would couple lifecycles. Cortex is tenant-facing, deserves its own ns. |
| Node group | `agent` | Tolerations + nodeSelector identical to `agent-will-smith`. No new node group needed. |
| TLS | ACM wildcard cert `*.cortex.mlytics.{com,co}` per env, DNS-validated | Extends existing `core/acm`. One cert per env account. |
| Pod Identity | One shared role per env (`cortex-uat`, `cortex-prod`), attached to both `cortex-web` and `cortex-api` ServiceAccounts | Two associations per env. See Section 2.6 for IAM policy. Splitting into per-service roles later is Terraform-only, no chart change. |
| Auth | Google OAuth → NextAuth in `cortex-web` issues session JWT → `cortex-api` validates per request via FastAPI auth/context dependencies | Already in app code per `cortex/CLAUDE.md`. Deploy just wires secrets + callback URL. |

### Per-environment URLs

| Env | Public host | OAuth callback | Wildcard cert | EKS account / cluster |
|---|---|---|---|---|
| UAT | `cortex.mlytics.co` | `https://cortex.mlytics.co/api/auth/callback/google` | `*.cortex.mlytics.co` | `147997115496` / `uat-data-platform-eks` (`ap-northeast-1`) |
| PROD | `cortex.mlytics.com` | `https://cortex.mlytics.com/api/auth/callback/google` | `*.cortex.mlytics.com` | `241533128494` / `prod-data-platform-eks` (`ap-northeast-1`) |

`mlytics.co` is the company's UAT root (per user direction). `mlytics.com` stays the public brand for PROD. The wildcard certs differ per env because the apex differs — they cannot be shared.

### Why these choices

- **Web-only public, api in-cluster** — minimal attack surface; api never exposed publicly. Flipping to public later costs ~2–4h (one ingress block + one cert SAN + CORS values), no app code, because the chart's ingress is `{{- if .Values.ingress.hosts }}`-gated already (per `agent-will-smith` pattern).
- **Wildcard cert up-front** — adds zero recurring cost; saves a future Terraform/registrar round-trip every time we want a new subdomain.
- **Dedicated `cortex` namespace** — independent RBAC, NetworkPolicies, and resource quotas later. Decouples from `agent-will-smith` blast radius.

## Section 2 — IaC scope (`data-platform-iac` changes)

A new sibling domain `agents/` is introduced. Existing `aigc/` is **not touched** — its `agent-will-smith` infra stays where it is. The migration of `aigc/*` → `agents/*` is a separate exercise, out of scope for this go-live.

### 2.1 `agents/ecr` — **NEW MODULE**

Mirror of `aigc/ecr` shape (FinOps module, repo policy, lifecycle policy for non-prod). `local.ecr_repositories`:
```hcl
cortex_api = "agents/cortex-api"
cortex_web = "agents/cortex-web"
```
Apply in UAT account (`147997115496`) and PROD account (`241533128494`). Non-prod keeps last-50 lifecycle rule; PROD does not.

### 2.2 `core/acm` — extend

New `aws_acm_certificate` resources, DNS validation:
- UAT: `*.cortex.mlytics.co` (SAN: `cortex.mlytics.co`)
- PROD: `*.cortex.mlytics.com` (SAN: `cortex.mlytics.com`)

Terraform outputs the validation CNAME(s); spec documents the registrar-paste step. Re-apply once validation completes to flip cert status to ISSUED.

### 2.3 `agents/rds` — **NEW MODULE**

- Engine: Postgres 16 (pinned via `~>`).
- Module: `terraform-aws-modules/rds/aws` (`~> 6.x`).
- UAT: `db.t4g.medium`, single-AZ, gp3 20 GiB, 7-day automated backup.
- PROD: `db.t4g.large`, multi-AZ, gp3 50 GiB, 30-day automated backup, deletion protection on.
- Subnets: private subnets from `data.terraform_remote_state.vpc`.
- SG: ingress `5432` only from EKS node SGs.
- Auto-gen 32-char password → `cortex/<env>/rds/credentials` SecretsManager entry as `{username, password, host, port, dbname}` JSON.
- FinOps tags: `finops-workload_name = "aws:rds:cortex"`, `finops-customer_name_id = "agents"`.

### 2.4 ~~`agents/elasticache`~~ — **DEFERRED**

Not provisioned in this go-live. App-side fallback (see Section 1 locked-in decision #3) keeps cortex-api functional. Module can be added later without touching the chart, since the api will read its cache backend from a config var.

### 2.5 `agents/secretsmanager` — **NEW MODULE**

Empty entries with `lifecycle { ignore_changes = [secret_string] }`. Per-env names:

| Secret name | Initial value | Populator |
|---|---|---|
| `cortex/<env>/rds/credentials` | auto-gen JSON `{username, password, host, port, dbname}` | Terraform (via `agents/rds`) |
| `cortex/<env>/oauth/google` | empty `{"client_id":"","client_secret":""}` | Human (AWS console) |
| `cortex/<env>/nextauth/secret` | empty string | Human (`openssl rand -base64 32`) |
| `cortex/<env>/databricks/pat` | empty `{"host":"","token":"","http_path":""}` | Human (Databricks workspace) |

### 2.6 `agents/pod-identity` — **NEW MODULE**

One shared role per env, attached to both `cortex-web` and `cortex-api` ServiceAccounts via two Pod Identity associations. Role name: `cortex-<env>` (e.g. `cortex-uat`, `cortex-prod`). Two associations per env → two roles total.

Per-env IAM policy:
- `secretsmanager:GetSecretValue` scoped to the four secret ARNs in 2.5.
- `secretsmanager:DescribeSecret` on the same ARNs (CSI driver needs this).

Why one shared role: cortex-web has no AWS calls today, so a dedicated empty-policy role would be ceremony with no benefit. If web ever needs distinct permissions, splitting later is a Terraform-only change with no chart impact.

### 2.7 `agents/github-oidc` — **NEW MODULE**

GitHub Actions OIDC trust + role for `cortex`'s `deploy.yml`. Created per env account (UAT `147997115496`, PROD `241533128494`).

Resources per account:
- `aws_iam_openid_connect_provider` for `token.actions.githubusercontent.com` (create if not present in the account; check first — the provider is account-wide, may already exist).
- `aws_iam_role` named `cortex-github-actions-deploy` with assume-role policy restricted to:
  - `repo:mlytics/cortex:ref:refs/heads/develop` (UAT account)
  - `repo:mlytics/cortex:ref:refs/heads/main` (PROD account)
  - audience `sts.amazonaws.com`.
- `aws_iam_role_policy` scoped to ECR push on the two repos only:
  - `ecr:GetAuthorizationToken` on `*` (required, account-scoped action).
  - `ecr:BatchCheckLayerAvailability`, `ecr:InitiateLayerUpload`, `ecr:UploadLayerPart`, `ecr:CompleteLayerUpload`, `ecr:PutImage`, `ecr:BatchGetImage` on the two ECR repository ARNs from `agents/ecr` outputs.

Role ARN is set as the `AWS_DEPLOY_ROLE_ARN` secret in the `cortex` GitHub repo, scoped per **GitHub Environment** (`uat` and `prod` Environments — `develop` branch uses `uat`, `main` branch uses `prod`). This also lays groundwork for the future CD automation follow-up (Option C) which needs Environments for manual approval gates on PROD.

### 2.8 App-side TODO (out of IaC, in `cortex/api`)

- Add `CORE_MEMBERSHIP_CACHE_BACKEND` config field defaulting to `memory`.
- Implement `MembershipCacheMemory` (LRU + 60s TTL) alongside the existing Redis-backed cache.
- Wire selection in the InfraContainer.
- Test: existing membership tests should pass with `backend=memory`.

### Order of apply (per env)

1. `agents/ecr` (independent).
2. `agents/github-oidc` (depends on `agents/ecr` outputs for the ECR ARN allowlist). Unblocks cortex CI push.
3. `core/acm` (independent; pauses on registrar CNAME paste before re-apply).
4. `agents/rds` (depends on `core/vpc` remote state — already exists).
5. `agents/secretsmanager` (RDS-credentials entry depends on 2.3 output; empty entries independent).
6. `agents/pod-identity` (depends on secret ARNs from step 5).

### Explicitly out of scope

- ElastiCache Redis (deferred, per user direction).
- WAF / Shield (revisit if api goes public later).
- CloudFront (revisit if TTFB demands it).
- New VPC / EKS / node group (`agent` node group accommodates cortex pods).
- Renaming `aigc/` → `agents/` for existing `agent-will-smith` infra (separate exercise).

## Section 3 — Helm chart scope (`data-platform-helm-charts` changes)

### 3.1 Chart organization — one umbrella

`charts/cortex/` is a single umbrella chart with two subcharts (`cortex-web`, `cortex-api`). Single release per env. Both services version-locked to the same `cortex` repo git SHA — release the pair together. Aligns with the umbrella migration direction documented in `data-platform-helm-charts/CLAUDE.md` (monitoring/keda are already umbrellas).

### 3.2 Layout

```
charts/cortex/
├── Chart.yaml                 # apiVersion: v2; dependencies: cortex-web, cortex-api
├── values.yaml                # shared defaults (namespace, FinOps base, node selector, image registry)
├── templates/
│   ├── _helpers.tpl           # cortex.labels, cortex.selectorLabels, cortex.fullname
│   ├── namespace.yaml         # creates `cortex` ns (gated on .Values.createNamespace)
│   ├── serviceaccount.yaml    # one SA per service, both labeled for the shared cortex-<env> Pod Identity role
│   └── secretproviderclass.yaml  # CSI driver class referencing the four cortex/<env>/* secrets
└── charts/
    ├── cortex-web/
    │   ├── Chart.yaml
    │   ├── values.yaml
    │   └── templates/
    │       ├── deployment.yaml     # port 3000, Next.js standalone, readiness=/api/health
    │       ├── service.yaml        # ClusterIP, port 3000
    │       ├── ingress.yaml        # ALB, public host, ACM cert from values
    │       └── configmap.yaml      # NEXTAUTH_URL, NEXT_PUBLIC_*, internal api URL
    └── cortex-api/
        ├── Chart.yaml
        ├── values.yaml
        └── templates/
            ├── deployment.yaml     # port 8000, FastAPI, readiness=/health/ready
            ├── service.yaml        # ClusterIP, port 8000 (no ingress)
            └── configmap.yaml      # DB host, log config, Databricks workspace URL
```

### 3.3 Secret mounting (SecretProviderClass at the umbrella level)

Both subcharts mount the same `cortex/<env>/*` secrets, so the CSI `SecretProviderClass` is defined once at the umbrella level. The class's `secretObjects:` block syncs to a K8s Secret that pods consume via `envFrom.secretRef`. Subcharts reference the class by name.

### 3.4 Per-env values shape

Each env file `environments/<env>/cortex/values.yaml` sets only what differs. Example (UAT):

```yaml
global:
  aws:
    accountId: "147997115496"
    region: ap-northeast-1
    env: uat
  domain: cortex.mlytics.co
  certificateArn: arn:aws:acm:ap-northeast-1:147997115496:certificate/<uuid>

mlytics:
  labels:
    finops-env: uat

cortex-web:
  image:
    repository: 147997115496.dkr.ecr.ap-northeast-1.amazonaws.com/agents/cortex-web
    tag: "<git-sha>"
  replicaCount: 2
  ingress:
    hosts:
      - host: cortex.mlytics.co
        paths: [{path: /, pathType: Prefix}]
    annotations:
      alb.ingress.kubernetes.io/scheme: internet-facing
      alb.ingress.kubernetes.io/listen-ports: '[{"HTTPS":443}]'
      alb.ingress.kubernetes.io/ssl-redirect: "443"
      alb.ingress.kubernetes.io/certificate-arn: "<from global.certificateArn>"
      alb.ingress.kubernetes.io/healthcheck-path: /api/health
  env:
    NEXTAUTH_URL: https://cortex.mlytics.co
    CORTEX_API_URL: http://cortex-api.cortex.svc.cluster.local:8000

cortex-api:
  image:
    repository: 147997115496.dkr.ecr.ap-northeast-1.amazonaws.com/agents/cortex-api
    tag: "<git-sha>"
  replicaCount: 2
  env:
    CORE_MEMBERSHIP_CACHE_BACKEND: memory
    CORE_LOG_LEVEL: info
```

### 3.5 Parameterized probes (avoid the `agent-will-smith` foot-gun)

The sibling chart hardcodes `port: 8000` and `path: /health` in its `ingress.yaml`. Cortex has two services with different ports + paths, so the new chart parameterizes:

```yaml
# cortex-web/values.yaml defaults
service: { port: 3000 }
probes:
  readinessPath: /api/health
  livenessPath: /api/health

# cortex-api/values.yaml defaults
service: { port: 8000 }
probes:
  readinessPath: /health/ready
  livenessPath: /health/live
```

Both flow into deployment.yaml and the ALB `alb.ingress.kubernetes.io/healthcheck-path` annotation so values cannot drift.

### 3.6 Required conventions from helm-charts CLAUDE.md

| Convention | Where applied | Notes |
|---|---|---|
| FinOps labels | `_helpers.tpl` for both subcharts | `finops-customer_name_id: agents`, `finops-workload_name: aws_eks_cortex_<web|api>`, `finops-business_criticality: high` (PROD) / `medium` (UAT). |
| Node group isolation | Deployment templates | `nodeSelector: {nodegroup: agent}`, `tolerations: [{key: dedicated, value: agent, effect: NoSchedule}]`. |
| Resource req/limits | Both deployments | Requests `cpu: 200m / memory: 512Mi`; memory limit `1Gi`; no CPU limit. PROD values override upward. |
| Image tag strategy | Per-env values, not chart defaults | Tags are git SHA from `cortex/` build; `latest` never used in env values. Chart default tag empty to force explicit setting. |
| ConfigMap checksum annotation | Both deployments | `checksum/config: {{ include ... \| sha256sum }}` so config edits trigger a rollout. |

### 3.7 Makefile target

```make
helm-cortex:
	helm upgrade --install cortex charts/cortex \
		-f environments/$(ENV)/cortex/values.yaml \
		--namespace cortex --create-namespace
```

### 3.8 Explicitly out of scope for go-live

- **KEDA** — cortex services are request-scale, not queue-scale. The repo's KEDA convention exists for SQS-triggered workloads; HTTP services don't fit. HPA can be added later if traffic data justifies it.
- **HorizontalPodAutoscaler** — deferred until baseline traffic exists.
- **PodDisruptionBudget** — worth adding to PROD values later; not blocking go-live.
- **NetworkPolicy** — deferred; add when other teams share the `agent` ns or when api goes public.

## Section 4 — CI / release flow

### 4.1 Today's state

`cortex/.github/workflows/deploy.yml` runs on push to `develop`/`main`, builds both Dockerfiles, pushes to ECR with two tags (`<sha>` and `latest`), and prints a TODO to bump the image tag in the helm-charts repo. No automated cluster apply.

### 4.2 Target state — promote-the-SHA

Image SHA is the unit of promotion. The exact bytes that ran in UAT run in PROD. No second build for PROD.

```
cortex repo                                helm-charts repo                       EKS
─────────────                              ────────────────                       ───
push to develop                            ┌─ release/<sha>-uat PR ─┐
  ► build + push                ─────────► │ bumps:                 │ ──► human runs:
    agents/cortex-api:<sha>                │   cortex-web.tag=<sha> │      make helm-cortex ENV=uat
    agents/cortex-web:<sha>                │   cortex-api.tag=<sha> │
                                           └─────────────────────────┘

push to main (after UAT validation)        ┌─ release/<sha>-prod PR ─┐
  (no rebuild — same image)                │ same diff, prod values   │ ──► human runs:
                                           └──────────────────────────┘      make helm-cortex ENV=prod
```

### 4.3 Required edits to `cortex/.github/workflows/deploy.yml`

```diff
 env:
   AWS_REGION: ap-northeast-1
-  ECR_REPO_API: aigc/cortex-api
-  ECR_REPO_WEB: aigc/cortex-web
+  ECR_REPO_API: agents/cortex-api
+  ECR_REPO_WEB: agents/cortex-web
```

Plus: **drop the `:latest` tag push.** The helm-charts CLAUDE.md explicitly warns against `:latest`; keeping it pushed encourages footguns. Tag with `<sha>` only:

```diff
-          docker build -t "$ECR_REGISTRY/$ECR_REPO_API:$IMAGE_TAG" -t "$ECR_REGISTRY/$ECR_REPO_API:latest" api/
-          docker push "$ECR_REGISTRY/$ECR_REPO_API:$IMAGE_TAG"
-          docker push "$ECR_REGISTRY/$ECR_REPO_API:latest"
+          docker build -t "$ECR_REGISTRY/$ECR_REPO_API:$IMAGE_TAG" api/
+          docker push "$ECR_REGISTRY/$ECR_REPO_API:$IMAGE_TAG"
```
(Same diff for `cortex-web`.)

Plus: `AWS_DEPLOY_ROLE_ARN` secret in the GitHub repo points at the role created by `agents/github-oidc` (Section 2.7). One value per env (use GitHub Environments, or distinct secret names — to be decided in the implementation plan).

### 4.4 Release PR mechanics in `data-platform-helm-charts`

For each env cutover:

1. **Title:** `Release cortex <sha> to <env>`.
2. **Diff:** minimal — only `environments/<env>/cortex/values.yaml` bumps `cortex-web.image.tag` and `cortex-api.image.tag`.
3. **PR body** includes:
   - Link to `cortex` repo commit range.
   - Smoke-test checklist (Section 6).
   - Rollback instructions.
4. **Merge → reviewer runs `make helm-cortex ENV=<env>`** locally with the correct `kubectl` context (Option B explicitly accepts the manual-apply pattern; CD automation is a follow-up — see Option C in brainstorm).

### 4.5 Rollback

```bash
helm rollback cortex -n cortex                 # immediate revert to previous revision
git revert <release-pr-merge-sha>              # restore git ↔ cluster correspondence
# new PR merges, re-run make helm-cortex ENV=<env>
```

Schema migrations (Alembic) must be backwards-compatible with N-1 — release additive migrations first, then code that uses them. Discipline lives in the cortex repo, not in this deploy plan.

### 4.6 Not in this go-live

- No Argo CD / GitOps controller (reuse `agent-will-smith`'s manual-apply pattern).
- No floating `:uat` / `:prod` tags; SHA is the contract.
- No PR-preview environments.
- No `:latest` push.

## Section 5 — Secrets, DNS, OAuth prerequisites

### 5.1 Secrets that must be populated before `helm-cortex` runs

Per env, in AWS Secrets Manager (CSI mounts these into pods):

| Secret name | Source of value | Format | Who populates |
|---|---|---|---|
| `cortex/<env>/rds/credentials` | Terraform auto-gen | JSON `{username, password, host, port, dbname}` | Auto (no human action) |
| `cortex/<env>/oauth/google` | Google Cloud Console → OAuth 2.0 Client | JSON `{"client_id":"...","client_secret":"..."}` | Human, console |
| `cortex/<env>/nextauth/secret` | `openssl rand -base64 32` | string | Human, CLI |
| `cortex/<env>/databricks/pat` | Databricks workspace → Personal Access Token | JSON `{"host":"https://...","token":"...","http_path":"/sql/1.0/warehouses/..."}` | Human, Databricks |

Update via:
```bash
aws secretsmanager put-secret-value \
  --secret-id cortex/uat/oauth/google \
  --secret-string '{"client_id":"...","client_secret":"..."}' \
  --region ap-northeast-1
```
`lifecycle.ignore_changes = [secret_string]` prevents future Terraform applies from clobbering human-pasted values.

### 5.2 Google OAuth client setup (per env)

**Two clients, not shared.** Web app type. Confirmed: cortex-web uses NextAuth's Google provider, so the callback path is `/api/auth/callback/google`.

| Field | UAT | PROD |
|---|---|---|
| Client type | Web application | Web application |
| Authorized JS origins | `https://cortex.mlytics.co` | `https://cortex.mlytics.com` |
| Authorized redirect URIs | `https://cortex.mlytics.co/api/auth/callback/google` | `https://cortex.mlytics.com/api/auth/callback/google` |

After creation, paste `client_id` + `client_secret` into `cortex/<env>/oauth/google` SecretsManager entry.

### 5.3 DNS records (Cloudflare-side)

Registrar is Cloudflare. Terraform cannot create these; human pastes both steps.

**Step 1 — ACM validation CNAMEs.** After `core/acm` apply, Terraform outputs N validation CNAMEs (one per SAN). Paste into Cloudflare:

| Env | Record name | Record value | Type | Proxy status |
|---|---|---|---|---|
| UAT | `_<random>.cortex.mlytics.co` | `_<random>.acm-validations.aws` | CNAME | DNS-only (grey cloud) |
| PROD | `_<random>.cortex.mlytics.com` | `_<random>.acm-validations.aws` | CNAME | DNS-only (grey cloud) |

Re-apply Terraform after CNAMEs propagate (~5–15 min). Cert moves from `PENDING_VALIDATION` → `ISSUED`.

**Step 2 — Public host CNAME.** After `helm-cortex` apply, capture the ALB DNS name:
```bash
kubectl get ingress -n cortex cortex-web \
  -o jsonpath='{.status.loadBalancer.ingress[0].hostname}'
# k8s-cortex-cortexweb-abc123-789.ap-northeast-1.elb.amazonaws.com
```

| Env | Record name | Record value | Type | Proxy status |
|---|---|---|---|---|
| UAT | `cortex.mlytics.co` | `<alb-dns-name>` | CNAME | **DNS-only (grey cloud) — required** |
| PROD | `cortex.mlytics.com` | `<alb-dns-name>` | CNAME | **DNS-only (grey cloud) — required** |

**Cloudflare proxy gotcha (load-bearing):** Proxied (orange cloud) records make Cloudflare's edge terminate TLS with its own cert and re-handshake with the ALB, which breaks SNI against the ACM cert. The deploy is misconfigured if the cloud is orange. Future-proxying needs Cloudflare Origin CA or Full-Strict mode — out of scope.

### 5.4 Pre-cutover checklist (human prereqs, per env)

Before running `make helm-cortex ENV=<env>`:

- [ ] All Section 2 Terraform modules applied for `<env>` account.
- [ ] ACM cert `<env>` shows status `ISSUED` in AWS console.
- [ ] Google OAuth client `<env>` created; `client_id`/`client_secret` pasted into SecretsManager.
- [ ] `NEXTAUTH_SECRET` `<env>` generated and pasted.
- [ ] Databricks PAT `<env>` generated and pasted with `host`, `token`, `http_path`.
- [ ] `cortex` repo `deploy.yml` PR merged (ECR paths `agents/*`, `:latest` push removed, OIDC role ARN secret set).

### 6.4 Rollback triggers + actions

| Trigger | Action |
|---|---|
| Pods `CrashLoopBackOff` after install | `helm rollback cortex -n cortex` (or fix-forward on first install — no previous revision). |
| Sign-in fails with `redirect_uri_mismatch` | OAuth client redirect URI doesn't match `NEXTAUTH_URL` + `/api/auth/callback/google`. Fix in Google Console; no helm rollback. |
| Sign-in works but api returns 401 | JWT issuer mismatch between web and api. Reconcile `CORE_AUTH_*` env values; patch + re-apply. |
| Ingress stays `<pending>` | AWS Load Balancer Controller RBAC. Check controller logs. (Controller is pre-existing.) |
| Cloudflare 525 or browser cert warning | Record is Proxied (orange cloud). Flip to DNS-only. |
| DB connection refused | RDS SG ingress missing EKS node SGs. Patch `agents/rds` + re-apply. |

### 6.5 Definition of done + bake

Per env, all of:
- Section 6.1 phases completed.
- Section 6.3 smoke checks pass end-to-end.
- A second human (not the deployer) has signed in and confirmed the dashboard renders their org.

**Bake:** UAT for 2 business days before promoting the same SHA to PROD. Shorter than a calendar week — enough to catch obvious regressions across two business days of casual use; not so long that PROD promotion drags. Extend ad-hoc if anomalies surface.

### 6.6 Effort estimate

Rough, optimistic — actual time likely shorter given the `agent-will-smith` chart is a close template and the IaC modules use `terraform-aws-modules/*` heavily:

| Phase | Effort |
|---|---|
| Section 2 IaC (UAT) | ~1.5 dev-days |
| Section 3 Helm chart authoring (umbrella + 2 subcharts + UAT values) | ~1 dev-day |
| App-side membership cache backend | ~½ dev-day |
| UAT cutover + smoke | ~½ dev-day, plus ~1 day calendar (DNS propagation, OAuth setup) |
| PROD repeat (values file + apply, no chart/IaC re-author) | ~½ dev-day |
| **Total focused work** | **~4 dev-days** + bake calendar time |

If anything blows the budget, the usual suspect is **DNS validation calendar lag** (Cloudflare propagation, then a second Terraform apply for ACM, then another wait for ALB DNS to be live). The work itself is small; the waiting is the variable.

## Closing notes

This spec is a deploy plan, not a re-design of cortex. It assumes the cortex app code already works locally end-to-end (Google sign-in → dashboard renders → api returns tenant-scoped data). The deploy makes it reachable; it doesn't change app contracts.

Two follow-ups deliberately deferred from this go-live, queued for after PROD baselines:

1. **CD automation (Option C from brainstorm).** Once one PROD release has happened by hand, extend `cortex/.github/workflows/deploy.yml` to also do the helm apply (with manual approval gate for PROD). Eliminates the "release PR + manual `make`" loop.
2. **`aigc/` → `agents/` migration.** Move existing `agent-will-smith` infra into the new `agents/` domain so cortex and agent-will-smith share one namespace. Pure Terraform state-mv work, no downtime if done carefully.
